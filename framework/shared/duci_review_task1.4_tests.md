## 2. 覆盖率审查 (核心边界检查)
织巢编写的测试用例仅涵盖了“快乐路径”（Happy Path），严重缺失防御性测试。以下核心边界未被覆盖：

### 2.1 软删除 (`test_workspace_soft_delete.py`)
- [ ] **路径不存在**: 尝试删除一个不存在的文件时是否抛出异常？
- [ ] **权限不足**: 当文件只读或无写权限时，软删除是否失败？
- [ ] **重复删除**: 已经处于 `.trash` 中的文件再次执行软删除的行为？

### 2.2 恢复 (`test_workspace_restore.py`)
- [ ] **恢复不存在的文件**: 尝试从 `.trash` 恢复一个不存在的文件？
- [ ] **目标冲突**: 恢复时，原始位置已存在同名新文件，如何处理？（覆盖/报错/重命名？）
- [ ] **权限冲突**: 原始目录无写入权限时的恢复行为？

### 2.3 彻底删除 (`test_workspace_permanent_delete.py`)
- [ ] **非回收站文件**: 严禁通过此机制彻底删除 `.trash` 目录以外的文件。目前的测试用例 (`test_trash_status_check`) 仅做了字符串匹配，未进行实际的拦截测试。
- [ ] **彻底删除不存在的文件**: 尝试删除不存在的回收站文件是否幂等？

## 3. 逻辑一致性审查 (符合《阿尔法 P1 修复方案》)
- 方案要求 `.trash` 作为工作区内的隐藏目录。目前测试仅模拟了移动操作，未体现“安全”二字。
- 缺乏对符号链接（symlinks）处理的测试，这在工作区管理中极易导致逻辑漏洞。

## 4. 改进建议
- 引入 `pytest.raises` 捕获预期异常。
- 模拟文件系统异常情况（如权限冲突）。
- 明确彻底删除的边界逻辑，增加“非法路径拦截”测试用例。
