## 📋 一、需求深度分析

### 1.1 园长真实意图洞察
园长提出的“大家一起的聊天室”，表面上是技术实现，实质上是**协作文化的重大升级**：
- **从异步到实时**：当前 Event Bus 是异步通信，园长希望看到**实时对话**。
- **从孤岛到协同**：当前工作流分散在各处，园长希望**全链路可视化**。
- **从被动到主动**：当前需要人工触发，园长希望**自动流转**（A完成→主动@B）。

### 1.2 核心价值主张
1. **协作透明化**：让整个动物园的工作流程“看得见、摸得着”。
2. **责任明确化**：通过 @ 机制，明确每个环节的责任人。
3. **效率最大化**：减少等待时间，实现无缝交接。

### 1.3 优先级判断
- **战略价值**：极高（协作文化升级）。
- **紧急程度**：低（园长明确表示“优先级不高”）。
- **技术复杂度**：中等（基于现有基础设施）。
- **建议等级**：**P3（低优先级，但有长期价值）**。

## 🏗️ 二、技术架构设计

### 2.1 整体架构
```
前端聊天界面 (Chat UI) (基于现有研发中心V2.8扩展)
实时通信层 (WebSocket/SSE) (可选：若需要真正的实时推送)
Event Bus 适配层 (将聊天消息转换为事件，维持现有架构)
现有 Event Bus 基础设施 (刚刚完成，支持异步通信)
```

### 2.2 核心组件设计

#### 2.2.1 聊天室前端
- **技术栈**：在现有研发中心 (`http://localhost:18792`) 基础上扩展。
- **界面布局**：左侧现有看板，右侧新增聊天面板。
- **消息类型**：
  - 普通消息（园长、Panda、各成员发言）
  - 需求消息（带格式化，可触发工作流）
  - @ 消息（触发特定成员处理）
  - 状态消息（工作流流转通知）

#### 2.2.2 工作流引擎
- **核心逻辑**：监听特定事件，自动触发下一步。
- **状态机**：
  需求下发 → 阿尔法分析 → @毒刺审计 → @织巢编码 → @史官归档
- **主动@机制**：通过 Event Bus 发布特定类型事件，触发对应成员的子智能体唤醒。

#### 2.2.3 Event Bus 集成
- **消息转换**：聊天消息 → 标准化事件。
- **事件类型**：
  - `chat.message`：普通聊天
  - `workflow.trigger`：工作流触发
  - `member.mention`：成员提及
  - `task.transfer`：任务交接

### 2.3 数据模型
```json
{
  "message": {
    "id": "msg_xxx",
    "type": "chat|workflow|mention",
    "sender": "panda|alpha|weaver|...",
    "content": "消息内容",
    "timestamp": "2026-04-06T21:30:00+08:00",
    "mentions": ["duci", "weaver"],
    "workflow_step": "analysis|audit|coding|archive"
  }
}
```

## ⚠️ 四、风险评估与应对策略

### 4.1 技术风险
- **实时性要求**：若需要真正的实时聊天，需引入 WebSocket/SSE，增加复杂度。
  - **应对**：初期使用轮询或长轮询，后续再升级。
- **Event Bus 性能**：高频聊天消息可能压垮 Event Bus。
  - **应对**：实现消息批处理与限流机制。
- **成员唤醒延迟**：@ 机制触发子智能体可能有 10-30 秒延迟。
  - **应对**：设置合理预期，并优化唤醒流程。

### 4.2 协作风险
- **消息过载**：所有成员都在一个聊天室，可能导致信息过载。
  - **应对**：实现频道/话题分组功能。
- **责任模糊**：多人同时 @ 可能引起混乱。
  - **应对**：清晰的交接协议与冲突解决机制。

### 4.3 安全风险
- **权限控制**：需要区分普通消息与工作流触发权限。
  - **应对**：基于角色的权限系统。
