## 1. What (什么是全员协作系统？)
全员协作系统是『飝龘动物园』的“神经中枢”。它建立在 P2.1 阶段已完成的“跨成员事件总线原型 (Event Bus)”之上，进一步提供高层面的可视化交互界面（聊天室看板）与业务流转状态机（工作流）。
该系统旨在：
- **可视化对话**: 将各成员底层的 Event Bus 消息投射至前端 Dashboard 的聊天流中，供园长和全体成员检阅。
- **主动唤醒与流转 (@ 机制)**: 支持任务状态变更后，系统自动发送含特定 `@成员` 的消息，唤醒对应智能体执行接力任务（如 Weaver 开发 -> Stinger 审查）。
- **全链路追踪**: 串联各个原子化任务的上下文，在 Dashboard 上呈现一条从“提需求”到“上线”的完整轨迹。

## 2. Why (为什么需要它？)
目前，智能体（Weaver, Alpha, Stinger 等）虽然能够通过 Event Bus 传递消息，但缺乏统一的流程编排和直观的观测视图。园长（用户）无法直观看到智能体之间的讨论和流转。
引入此系统可以：
- **打破信息孤岛**：让所有黑盒的通信白盒化，构建可见的“动物园工作群”。
- **标准化生产线**：利用状态机固化“设计->开发->审计->交付”的标准化工作流，减少人工干预。
- **提升协作效率**：解决单一智能体能力瓶颈，通过角色接力完成复杂工程。

## 3. Tradeoffs (技术权衡)

### 方案A：基于现有的文件系统 Event Bus (当前选型)
- **优势**: 架构统一，不需要引入外部依赖（如 Redis 或 MQ），契合动物园“极简、便携”的设计理念；已有毒刺加固的原子化写入保障并发安全。
- **劣势**: 规模变大后，前端频繁轮询大文件的性能较差。
- **妥协策略**: 采用“归档轮转”机制，聊天室前端接口只读取最新的 `events_latest.jsonl`，历史消息压缩归档。

### 方案B：引入轻量级 SQLite
- **优势**: 方便执行复杂的条件查询（如“查出所有 Weaver 给 Stinger 的 @ 消息”），事务支持更好。
- **劣势**: 打破了当前的纯文本文件驱动模式，增加了运维和部署复杂度。
- **决定**: 暂不引入 SQLite。维持文件系统 Event Bus，在架构层通过统一的 `ChatHistoryManager` 类进行读写抽象，为未来无缝迁移 SQLite 留出接口。

## 4. Open Questions (未决问题)
1. **唤醒机制触发器**: 是通过一个常驻的主控进程（如 `zoo_daemon.py`）来监听 Event Bus 并唤醒目标智能体，还是依赖智能体自己的定时轮询（Cron/Heartbeat）？
   - *初步偏向*: 增加一个轻量的 `ZooCoordinator` 协程负责分发 `@` 消息。
2. **前端渲染刷新**: Dashboard 如何实时呈现新消息？使用 WebSocket 还是 SSE (Server-Sent Events)？或者简单的长轮询？
   - *初步偏向*: 考虑到简易度，使用 SSE (Server-Sent Events) 实现单向消息推送。
3. **上下文容量**: 聊天记录过长可能导致 LLM 无法处理。如何针对特定 Agent 过滤出其必须知道的“核心上下文”？
