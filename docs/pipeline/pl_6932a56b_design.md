## 二、架构设计

### 2.1 What — 产出物

| 产出 | 路径 | 说明 |
|------|------|------|
| README.md | `/Volumes/data/workspace/code/feida_zoo/README.md` | 完整重写，含活跃成员、项目概述、结构、运行指南、Pipeline、截图 |
| Dashboard 截图 × 4 | `docs/pipeline/screenshot_*.png` | 看板/成员管理/聊天室/需求管理 → 复制到 README 可访问路径 |

### 2.2 Why — 为什么要这么改

| 当前问题 | 修改方案 | 理由 |
|----------|----------|------|
| 成员列表包含已退出成员 | 拆分「活跃成员」和「归档成员」 | 保留历史记录但不误导新读者 |
| 核心守则断裂 | 内联核心守则内容，删除空链接 | 信息必须自包含 |
| 无项目概述 | 开头增加项目简介段落 | README 最基本要素 |
| 无目录结构 | 用 tree 格式展示 | 帮助读者快速定位文件 |
| 无运行指南 | 新增依赖安装 + 启动命令 + 访问地址 | 让新成员能直接上手 |
| 无界面截图 | 嵌入 4 张 Dashboard 截图 | 需求明确要求，且视觉效果远胜文字 |
| 无 Pipeline 介绍 | 新增 Pipeline 工作流说明 | 这是项目核心协作机制 |
| 无技术栈 | 新增技术栈清单 | 帮助技术选型决策 |
| Dashboard 硬编码路径 | README 中使用 `${FEIDA_ZOO_HOME}` 环境变量说明 | 避免路径泄露 |

### 2.3 Tradeoff

| 选项 | 优点 | 缺点 | 选择 |
|------|------|------|------|
| 删除已退出成员信息 | 简洁 | 丢失历史信息 | ❌ |
| 拆分为活跃/归档 | 保留历史，不误导 | 略长 | ✅ |
| 截图用外部 URL | 不占用仓库空间 | 依赖外部服务 | ❌ |
| 截图存入仓库 | 永久可用，离线可看 | 增加仓库体积（约 600KB） | ✅ 4 张截图约 600KB，可接受 |
